Common Type System 
- Type Classification 
	- Value Types 
		- objects which hold values 
		- created on stack 
		- Inherit from System.ValueType which inherits from System.Object (root) 
		-> int i = 10; i is a reference to an integer object with value of 10 
		- BUilt-in types, Structs, Enums,
		- Promotable types - switch between stack and heap  
	- Reference Types
		- holds reference/pointers to memory locations 
		- heap location 
		- Class, Arrays, String, Interfaces, Delegates, Pointer 
		- always on the heap 
	
	int [] arr = new int[100]; 
	arr[0] = 10; ==> BOXING 
	==> creates int object 
	==> typecasts to system.object type 
	==> stores on the heap 
	==> gets the location ref and assigns to the array element 

	-> value type is converted/boxed into an object type and stored on the heap 
		-> the reference to that object is held by the array 
	array -> contiguous memory allocations which stores references/pointers to heap 			locations
	arr[0] holds the pointer of the 10 integer object created on the heap. 

	int x = arr[0]; ==> UNBOXING 
	
	obj.x => jump to the objects characteristics table - find x - get the address of x 
		jump to the address of x and extract the value 
	x => jump the location which holds the value

Arguments to functions 
1. Call By Value 	->  copy of the value is passed as new object. 
2. Call By Reference	-> copy of the address is passed 
			-> "ref" keyword is to be used at both actual and formal arg list 
			-> parameter must be initialized before usage 
			-> called function can change the parameter value 
3. Call By Out		-> function to generate a value and return the values 
			-> simulate multiple returns from function 
			-> use the "out" keyword in actual and formal arg list 
			-> out parameter need not be initialized 
			-> function has to initialize the out parameters before leaving

4. Call by params	-> vararg type of argument passing 
			-> 0 to 'n' number of values to a parameter 
			-> dynamically creates an array with the number of arguments passed
			-> "params" keyword in the argument list 
			-> only one params parameter is allowed in the argument list 
			-> params parameter should be the last argument in the list. 
5. Optional Arguments	-> define arguments with default values 
			-> allows to skip passing values to those arguments 
			-> if no values are passed, the default values are assigned 
6. Named Arguments	-> pass arguments by name rather than by position 
			-> allows to alter the argument passing sequence

Class in C# 
- can contain fields, methods, properties, indexers, events, constructors, destructors 
- is a ref type and inherits from System.Object 
- all class fields are initialized to their default values 
- can define parameterized or parameterless constructors or initializers 
- default parameterless constructor is provided by the runtime 
- can be inherited by other classes unless explicitly forbidden/prevented 
- class can inherit from one base at most, multiple inheritance is not supported 
- classes can implement multiple interfaces 
- all class members are private  by default. 

Constructors 
- are the first method to be invoked on the object, usually done by the CLR 
- used to initialize the newly created object 
- default parameterless constructor provided by the compiler 
- parameterized constructors can eb defined in code 
- can be overloaded - multiple constructors defined with differing arguments 
- can be chained into a constructor list - one ctor calling another ctor 

Object Model - Grady Booch -"OOAD with Applications" 
- Defines 7 elements which has to be supported by OO systems/languages 
- Major Elements (4) and Minor Elements (3) 
Major Elements should be compulsorily supported  by every OO language/system: 
1. Abstraction 		-> ability to extract information relevant to the problem domain 
2. Encapsulation 	-> hide the representation (memory) behind implementation (functions) 
				-> Accessor/Mutator methods, getter/setter methods, Properties 
				->System data should be accessible via functionalities (interfaces)
3. Hierarchy/Inheritance-> Ranking or Ordering of types 
				-> Every OO system should have exactly one root element
				-> A descendant element should have exactly one ancestor element (Single)
				-> Single and Multi-level should be supported 
				-> Multiple, Diamond, Hybrid are not to be supported 
4. Modularity		-> modular in nature, decomposed into smaller functional units 
				-> compartmentalization or Componentization should be supported 
				-> a component should ideally address one specific functionality of the app 
				
Minor Elements (3) - optional for OO languages to support these elements 
1. Concurrency		-> should support concurrent access to the object 
2. Typing		-> can be strongly typed or weakly typed. Strong typing is preferred
3. Persistence 		-> preserve the state of the object across time and space (Serialization)

Hierarchy/Inheritance 
- Any class in .NET is inheritable by default, 
- Re-use the code/share the code 
- All non-private members of the base class are accessible to the derived classes
- By default, the parameterless ctor of the base class is always invoked before the derived ctor runs
- Before creating the derived object, the base object is created first. 
- Derived class has an implicit pointer to the base object called "base" 

Polymorphic behavior 
- Same name multiple forms 
- Static Polymorphism 
	- overloading, compile time polymorphism, early binding 
	- the actual method to be invoked is known at compile time 
	- the number, order or type of parameters must differ 
	- the visibility and the return type of the method are not considered 
- Dynamic Polymorphism
	- the actual method to be invoked is known at runtime using virtual tables [VTABLES]
	- the method signature cannot be changed by the derived class 
	- visibility, return type, name, argument list cannot change - implementation can 
	- Overriding, Late-Binding or Runtime Polymorphism

Prevent inheritance - use the "sealed" keyword. 
Sealed keyword on 
class - prevent inheritance of the class 
method - prevents further overriding of the method 

Abstract 
class 
	- full fledged class that contains fields, methods, properties, constructors, destructors,
	- it also contains zero or more unimplemented methods - signature is declared 
	- the unimplemented methods have to be implemented using derived classes 
	- forcing inheritance to use the abstract class 
	- cannot be instantiated directly 

SOLID - 
S	SRP -> Single Responsibility Principle - each class should have one responsibility
O	OCP -> Open/Close Principle - a class should be closed for modification, open for extension
L	LSP -> Liskov Substitution P - replace derived references with base references with breaking
I	ISP -> Interface Segregation P - create smaller interfaces rather than one massive interface
D	DIP -> Dependency Inversion P - Depend on abstraction rather than concrete implementations

methods 
	- are unimplemented methods - with only the declaration provided 
	- pure virtual methods 
	- derived class should use the override keyword to implement the abstract method 
	- are implicitly virtual 
	- cannot be declared as static, sealed or private
